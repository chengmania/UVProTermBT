2026-07-19 — SSTV M3 TX CRACKED (radio keys + transmits our audio)
- Definitive method: ran HTCommander's Linux desktop build on this box, captured
  the FULL HCI with `sudo btmon -w` (untruncated, unlike the phone's filtered
  btsnooz), and diffed its working transmit against ours. Two byte-level bugs:
  1) SBC allocation: sbc.py had SBC_AM_LOUDNESS=0x01 (backwards); Loudness is 0x00
     (SNR is 0x01). We were sending SNR-allocated SBC (header 9c 73...); the radio
     wants Loudness (9c 71...) and silently discarded our audio. Fixed.
  2) Audio frame size: HTCommander sends ~352-byte frames (~4 SBC frames of 88B)
     every ~15ms; we crammed 50 SBC frames (~4400B) into one 0x7e frame. The radio
     ignores oversized frames. transmit_pcm now emits frame_length*4 packets.
- Result: `audio_tx --tone --control` → radio htStatus flips to c0 (TX active) at
  t~0.3s and HOLDS for the whole tone; Greg heard it on his IC-9700. TX confirmed.
- So the full SSTV pipeline is now achievable both ways: RX (proven earlier) and
  TX (now). Keying is implicit (audio stream keys the radio) with the GAIA control
  channel open (--control does the handshake). No PTT command needed.

  False, and _after_attach took the browser fallback. (Earlier verification
  imported pat_panel BEFORE QApplication, so it missed this.)
- Fix: __main__._enable_webengine_gl_sharing() sets AA_ShareOpenGLContexts before
  QApplication (called right after the lib-path shim, before the app/wizard).
  Attribute-only — no eager ~100 MB Chromium import; the WebEngine import stays
  lazy for users who never open Winlink, and now succeeds so PAT embeds.
- Verified in the real app order (LD path set as the shim produces): QApplication
  after the attribute -> lazy pat_panel import -> WEBENGINE_AVAILABLE True,
  MainWindow._pat_panel._view is a real QWebEngineView (VERDICT: EMBEDS-IN-WINDOW).
  Without the attribute: False / NoneType (reproduced the bug).

  app (as main() does) + test_gl_sharing_attribute_enabled asserts the attribute

2026-07-13 — Phase 2 (RFCOMM link) — root cause found and fixed
- User found two independent real-world writeups documenting a working
  classic-BT KISS-TNC setup for this exact radio family (VR-N76/BTech
  UV-Pro) on Linux: a HamRadioTech article (DC6AP, Sep 2025, "VR-N76
  (UV-PRO) with KISS-TNC under Linux") and the
  islandmagic/kiss-tnc-test Raspberry Pi guide. Both independently
  confirm RFCOMM channel 1 (matching our own khusmann/benlink-informed
  finding), and both call out disabling BlueZ's Bluetooth headset/
  Hands-Free profile as a required fix we'd never tried.
- Confirmed the root cause directly on this machine via
  `journalctl -u bluetooth`: repeated, exactly-timed log lines
  throughout every flaky session tonight —
  `bluetoothd[1256]: src/profile.c:ext_connect() Hands-Free unit failed
  connect to 38:D2:00:01:38:8F: Connection refused (111)`. BlueZ's
  `hfp-hf` (Hands-Free unit/client role) plugin auto-connects to the
  radio's advertised Handsfree Audio Gateway service (SDP `Voice
  Gateway`/0x111f, channel 3), contending with our RFCOMM channel-1 KISS
  connection attempts at the BlueZ/kernel level, regardless of which
  client API initiates the connection.
- The two articles' own fix (`/etc/bluetooth/input.conf`,
  `Disable=Headset`) is stale for this system's BlueZ 5.72 — confirmed
  that option doesn't exist in this version's input.conf (HID-only).
  Found the modern equivalent: `bluetoothd --help` documents
  `-P/--noplugin=NAME`; `strings /usr/libexec/bluetooth/bluetoothd`
  confirmed compiled-in plugin names `hfp-hf`/`hfp-ag`/`hsp-hs`/`hsp-ag`.
- APPLIED (user-confirmed, system-wide change): created
  `/etc/systemd/system/bluetooth.service.d/override.conf` disabling only
  `hfp-hf` (the exact role in the failing log line):
  ```ini
  [Service]
  ExecStart=
  ExecStart=/usr/libexec/bluetooth/bluetoothd --noplugin=hfp-hf
  ```
  Applied via `sudo systemctl daemon-reload && sudo systemctl restart
  bluetooth`; verified new bluetoothd process running with the flag.
  A2DP (music/audio playback) endpoints still register normally —
  unaffected. Rollback: remove the override.conf, daemon-reload, restart.
- Updated pairing procedure to include `trust <MAC>` after `pair <MAC>`
  (both external articles use it, we never had) — documented in
  docs/PROTOCOL.md §2 and scripts/monitor.py's docstring.
- Architecture decision: keep RfcommKissLink's raw
  socket.AF_BLUETOOTH approach rather than switching to `rfcomm bind` +
  pyserial (which both external guides use) — the root cause was at the
  BlueZ/kernel level below either client API, so their reliability is
  attributable to the headset fix, not to `rfcomm bind` itself. No
  proven benefit to switching; would add a sudo-gated system device
  node, a new pyserial dependency, and a duplicate reconnect lifecycle.

     the very link it just configured. FIX: bt.py uses PRIVATE bus connections
     (dbus.SystemBus(private=True), closed after use) so it never pollutes the
     singleton the link relies on. Reproduced the failure and confirmed the fix.

2026-07-16 — Phase C: Winlink via PAT (KISS-over-TCP bridge)
- Greg's question — "can we AGWPE with a KISS TNC over BT rather than a BT
  serial connection?" — led to a much simpler architecture than the original
  AGWPE-server idea. Found that pat-gensio (PAT with built-in AX.25) connects
  to a KISS TNC over TCP directly (gensio `kiss,tcp,localhost,8001`). So we
  don't implement AGWPE OR do AX.25 for Winlink — PAT does it all; we just
  re-serve the radio's KISS stream on a local TCP socket.
- uvprotermbt/kiss_tcp.py: KissTcpServer — a transparent byte pipe on top of
  the existing BT link. Background accept/read loop forwards PAT→radio via
  link.send(); the GUI feeds radio→PAT via feed_from_radio() from its RX poll.
  One client (one radio), thread-safe, no GUI-thread I/O.
- Winlink tab: Start/Stop Winlink Bridge button + the exact PAT config string
  (kiss,tcp,127.0.0.1,8001). While bridging, the app relays RX to PAT, still
  decodes for the monitor, and BLOCKS its own TX (PAT owns the radio). Poll
  detects PAT connect/disconnect. Stops on close.
- Chose KISS-over-TCP over AGWPE: ~120 lines vs a full AGWPE protocol + our own
  AX.25; PAT's mature stack (k>1) handles B2F better than our stop-and-wait.
- tests/test_kiss_tcp.py (3): both-direction relay, client-disconnect state,
  no-client no-op. Suite 39/39. GUI flow verified headlessly + rendered.
  docs/WINLINK_PAT.md: PAT/pat-gensio setup guide.
- NEXT (Greg, on-air): install pat-gensio, Start Winlink Bridge, point PAT at
  kiss,tcp,localhost:8001, connect to an RMS, send/receive a Winlink message.

2026-07-17 — Link reconnect fix (KeyError crash + connect/disconnect flapping)
- Symptom (Greg): on a fresh program start the radio wouldn't reconnect —
  appeared to connect then dropped, then locked in a connected/not-connected
  cycle; traceback showed KeyError "Can't register the object-path handler for
  '/org/bluez/uvprotermbt': there is already a handler".
- Root cause 1 (the KeyError / thread crash): every RfcommKissLink registered
  its _SerialProfile at the FIXED path /org/bluez/uvprotermbt on the process-
  wide shared SystemBus, and _run()'s teardown called UnregisterProfile but
  never _profile.remove_from_connection(). So Radio→Reconnect / Select Radio /
  MAC change → _rebuild_link() stops the old link and news up a second one →
  second _SerialProfile at the same path on the same bus → KeyError → link
  thread dies → link never reconnects.
- Fix 1: link.py now gives each instance a UNIQUE object path
  (/org/bluez/uvprotermbt/link<id>), and _run()'s finally block calls
  self._profile.remove_from_connection() (plus UnregisterProfile with the
  instance path). Unique path avoids the collision; releasing the handler is
  the correct teardown.
- Root cause 2 (the flapping): a fd HUP and an already-queued retry each armed
  their own _attempt_connect, which does Disconnect()-then-Connect() — two of
  them racing produce org.bluez InProgress and a connect/disconnect storm.
- Fix 2: reconnect timers now COALESCE — a single _reconnect_source id, guarded
  in _schedule_reconnect (no-op if one is pending), cancelled in
  _on_new_connection (we're up) and in _run() teardown. Added _cancel_reconnect.
- Root cause 3 (the REAL "KDE says connected but the app isn't"): the app used
  Device1.Connect() (blanket, all auto-connect profiles) + a whole-device
  Disconnect() first. But the UV-Pro also advertises Handsfree/Audio-Gateway,
  and BlueZ (radio Trusted + our profile AutoConnect) keeps the DEVICE connected
  via that audio profile even when our KISS/SPP channel is down. In that state
  Device1.Connect() is a no-op that never hands us the SPP fd → NewConnection
  never fires → KDE shows "connected" while the app sits at ○ BT. And the
  whole-device Disconnect() knocked out the audio profile too, fighting KDE and
  driving the flapping.
- Fix 3: drive OUR profile specifically — Device1.ConnectProfile(SPP_UUID)
  instead of Connect(), and drop the whole-device Disconnect(). ConnectProfile
  fires NewConnection with the RFCOMM fd even on an already-ACL-connected device.
  On a non-transient error we DisconnectProfile(SPP_UUID) (just our profile, not
  the device) to clear a stuck half-open SPP, then backoff-retry. Added a
  duplicate-fd guard in _on_new_connection (AutoConnect can race our
  ConnectProfile and offer a 2nd fd — keep the live one, close the dup).
- VERIFIED LIVE (2026-07-17, device already Connected via audio gateway, app
  closed): probe confirmed ConnectProfile fires NewConnection + delivers the fd;
  the patched RfcommKissLink connects in ~0.5s, repeatably across 3 back-to-back
  runs, with clean teardown (no KeyError, no stuck SPP).
- tests/test_link.py (new, 3): distinct profile paths; _schedule_reconnect
  coalesces to one timer; _on_new_connection cancels the pending reconnect and
  resets backoff. GLib faked (pure bookkeeping, no real loop). Suite 44/44.
- NEEDS ON-AIR (Greg): full app fresh start connects (● BT) and stays even when
  KDE already shows the radio connected; Radio→Reconnect no longer crashes the
  link thread; a wedged radio recovers without the flapping storm.
