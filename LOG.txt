2026-07-13 — Phase 2 (RFCOMM link) — implemented, channel corrected, still flaky live
- Added uvprotermbt/link.py: RfcommKissLink (socket.AF_BLUETOOTH/
  BTPROTO_RFCOMM), background RX thread + queue, send()/on_receive()/
  poll(), reconnect with backoff. Added scripts/monitor.py as a Phase-2
  hardware smoke test (connect, print raw + KISS-decoded frames).
- Confirmed via `sdptool records <addr>` (not `browse`, which comes back
  empty) from this machine directly: SDP shows RFCOMM channels 2 ("BS
  AOC"), 3 (Voice Gateway/HFP), 4 ("SPP Dev"/Serial Port 0x1101).
  Initially targeted channel 4 as the "proper" SDP-discoverable choice.
- CORRECTION: found the khusmann/benlink project on GitHub — a Python
  library specifically reverse-engineering this exact radio family
  (Benshi: BTech UV-Pro, RadioOddity GA-5WB, Vero VR-N76/N7500, BTech
  GMRS-Pro). It documents the real BLE control service
  (00001100-d102-11e1-9b23-00025b00a5a5 — not the 00000001-ba2a-... one
  we'd found, which is itself a red herring) and a GaiaFrame/Message
  command protocol. Its own RFCOMM example uses channel 1, not 4.
  Cross-checked against our earlier captured bytes: channel 4 traffic
  ends in `...ff 01 00` (a GaiaFrame header), channel 1 traffic starts
  with `c0 00` (raw KISS FEND + command byte, no envelope). Conclusion:
  channel 4 is a red herring (generic serial profile, not the real TNC
  path); channel 1 carries genuine raw KISS pass-through. Changed
  RfcommKissLink.DEFAULT_CHANNEL from 4 to 1. Full writeup in
  docs/PROTOCOL.md §2. Decided NOT to adopt benlink's Message/GaiaFrame
  protocol for our core KISS data path — our own kiss.py + RfcommKissLink
  on channel 1 is simpler and sufficient; benlink stays a good reference
  for future radio-control features (battery, channel switching).
- VERIFIED live from Linux (twice): a bare RFCOMM connect to channel 1
  succeeds cleanly after a fresh `bluetoothctl pair`. Neither test window
  happened to have live RF traffic, so we haven't yet decoded a real
  AX.25 frame end-to-end through this project — transport-level
  connection is confirmed, full data-path confirmation is still pending.
- NOT resolved: classic BT connection reliability. Spent a large amount
  of this session on live hardware debugging — saw clean connects,
  TimeoutError, ConnectionRefusedError, and BlueZ-level
  br-connection-refused (that last one specifically from an asymmetric
  bond: our side had removed its copy of the link key while the radio
  still remembered pairing with us — symmetric removal on both sides
  fixed that particular case). The radio's own "paired device" indicator
  was observed appearing and disappearing within a few seconds,
  independent of anything we did. A full power cycle (battery pull)
  restored BLE connectivity once when it had gone flaky. Holding a BLE
  connection open at the same time does NOT stabilize classic RFCOMM —
  confirmed independent. No access to the radio's own firmware logs, so
  further root-causing from software has hit its limit for tonight.
- NEXT: get a genuine live end-to-end KISS/AX.25 decode through
  RfcommKissLink on channel 1 (needs to coincide with actual RF traffic
  during a working connection window). If classic-BT flakiness keeps
  blocking progress, consider a dedicated session focused purely on

  with Phase 3 (AX.25 parsing) against real traffic — though Phase 3's

2026-07-13 — Phase 2 (RFCOMM link) — root cause found and fixed
- User found two independent real-world writeups documenting a working
  classic-BT KISS-TNC setup for this exact radio family (VR-N76/BTech
  UV-Pro) on Linux: a HamRadioTech article (DC6AP, Sep 2025, "VR-N76
  (UV-PRO) with KISS-TNC under Linux") and the
  islandmagic/kiss-tnc-test Raspberry Pi guide. Both independently
  confirm RFCOMM channel 1 (matching our own khusmann/benlink-informed
  finding), and both call out disabling BlueZ's Bluetooth headset/
  Hands-Free profile as a required fix we'd never tried.
- Confirmed the root cause directly on this machine via
  `journalctl -u bluetooth`: repeated, exactly-timed log lines
  throughout every flaky session tonight —
  `bluetoothd[1256]: src/profile.c:ext_connect() Hands-Free unit failed
  connect to 38:D2:00:01:38:8F: Connection refused (111)`. BlueZ's
  `hfp-hf` (Hands-Free unit/client role) plugin auto-connects to the
  radio's advertised Handsfree Audio Gateway service (SDP `Voice
  Gateway`/0x111f, channel 3), contending with our RFCOMM channel-1 KISS
  connection attempts at the BlueZ/kernel level, regardless of which
  client API initiates the connection.
- The two articles' own fix (`/etc/bluetooth/input.conf`,
  `Disable=Headset`) is stale for this system's BlueZ 5.72 — confirmed
  that option doesn't exist in this version's input.conf (HID-only).
  Found the modern equivalent: `bluetoothd --help` documents
  `-P/--noplugin=NAME`; `strings /usr/libexec/bluetooth/bluetoothd`
  confirmed compiled-in plugin names `hfp-hf`/`hfp-ag`/`hsp-hs`/`hsp-ag`.
- APPLIED (user-confirmed, system-wide change): created
  `/etc/systemd/system/bluetooth.service.d/override.conf` disabling only
  `hfp-hf` (the exact role in the failing log line):
  ```ini
  [Service]
  ExecStart=
  ExecStart=/usr/libexec/bluetooth/bluetoothd --noplugin=hfp-hf
  ```
  Applied via `sudo systemctl daemon-reload && sudo systemctl restart
  bluetooth`; verified new bluetoothd process running with the flag.
  A2DP (music/audio playback) endpoints still register normally —
  unaffected. Rollback: remove the override.conf, daemon-reload, restart.
- Updated pairing procedure to include `trust <MAC>` after `pair <MAC>`
  (both external articles use it, we never had) — documented in
  docs/PROTOCOL.md §2 and scripts/monitor.py's docstring.
- Architecture decision: keep RfcommKissLink's raw
  socket.AF_BLUETOOTH approach rather than switching to `rfcomm bind` +
  pyserial (which both external guides use) — the root cause was at the
  BlueZ/kernel level below either client API, so their reliability is
  attributable to the headset fix, not to `rfcomm bind` itself. No
  proven benefit to switching; would add a sudo-gated system device
  node, a new pyserial dependency, and a duplicate reconnect lifecycle.
  Documented as a fallback option if raw sockets are still unreliable
  after this fix.
- Polished uvprotermbt/link.py: differentiated ConnectionRefusedError/
  TimeoutError/generic OSError in _run()'s logging for actionable
  diagnostics (still same backoff behavior, no control-flow change);
  added a threading.Lock around _sock/_connected mutation in send()/
  _handle_disconnect() (defensive cleanup for a previously-benign race
  between the main thread's send() and the RX thread's own disconnect
  handling); send() failures are now logged instead of silent.
- VERIFIED: `pytest tests/` still 8/8 passing after the link.py changes.
- Documented direwolf's TX limitation with this radio (HamRadioTech:
  direwolf RX works, TX silently fails, cause unknown; recommends
  TFKISS) — noted for later Phase 4+ on-air TX testing, not installing
  new software this session since our current hardware test is RX-only.
- NEXT: live hardware re-test with the fix in place — re-pair with
  `remove` + `trust`, run scripts/monitor.py, confirm a connection that
  stays up (not flapping), and catch a real live KISS/AX.25 decode
  during actual RF traffic to finally close out the Phase 2 hardware
  test.

2026-07-14 — Phase 2 CLOSED: first live KISS/AX.25 decode; transport rewritten
- **Milestone: decoded live APRS traffic end-to-end for the first time.**
  Caught real Mic-E position beacons from the UV-Pro over Bluetooth:
  `KC3SMW-7 > <Mic-E dest> via WIDE1-1,WIDE2-1`, control 0x03 / PID 0xF0,
  raw KISS framing (`c0 00 ... c0`, no GaiaFrame envelope).
- **The raw-socket approach was a dead end — rewrote the transport.** The
  prior `RfcommKissLink` (raw `socket.AF_BLUETOOTH` to channel 1) never
  works with this radio: when the radio's BR/EDR is awake it *refuses*
  bare channel connects on every channel (swept 1–15), and it power-saves
  between pages so connects otherwise time out. Proved this session that
  the earlier "cat /dev/rfcomm0 held 60s = healthy link" reading was
  wrong — the open was just blocking on a connect that never completed
  (a direct socket connect test made it explicit: TimeoutError). Credit
  to Greg for pushing back with "I don't think you ever connected" — that
  was correct and redirected the whole session.
- **Root method (the fix): BlueZ SerialPort profile, not a raw socket.**
  The radio only serves KISS through its advertised SerialPort profile
  (UUID 0x1101) via an SDP-negotiated *profile* connection — same as YAAC
  on macOS and WoAD on Android. A groups.io YAAC/VR-N76 guide from Greg
  named the missing radio-side step (Enable KISS TNC) and confirmed the
  OS-mounts-a-serial-port model. Implemented on Linux by registering an
  `org.bluez.Profile1` (client role) over D-Bus and reading the RFCOMM fd
  BlueZ hands to `NewConnection`. Pure Python, no sudo, no `rfcomm` tool,
  no pyserial.
- **Two required prerequisites nailed down this session:**
  1. **Enable KISS TNC on the radio** (Settings → General Settings →
     KISS TNC → Enable KISS TNC). Not in the Rev3 PDF manual; it's a
     current-firmware feature. Until on, the profile connect is refused.
  2. **Clean symmetric bond.** The persistent pairing failure
     (`AuthenticationRejected` / `br-connection-refused`) was a stale link
     key in `/var/lib/bluetooth` that `bluetoothctl remove` silently
     skips when the device object isn't present. Fix: `sudo rm -rf` the
     device's bond dir + `systemctl restart bluetooth`, forget on the
     radio too, then pair+trust. Let KDE's bluedevil be the pairing agent
     (registering our own fails — one already exists).
- **Rewrote `uvprotermbt/link.py`** onto the profile method, keeping the
  exact public API (begin/stop/is_connected/send/on_receive/poll). Runs a
  GLib main loop on its own thread; `Disconnect()`-before-`Connect()` to
  clear `br-connection-busy`; treats `NoReply`/busy as transient (the
  `NewConnection` callback is the real success signal); backoff reconnect;
  reads the fd via a GLib IO watch into the existing rx queue. Guarded
  dbus/gi import so the package still imports (and codec tests still run)
  on hosts without the BlueZ bindings.
- **Rewrote `scripts/monitor.py`** as the reproducible smoke test (defaults
  to the UV-Pro MAC, prints decoded frames with a source>dest-via-path
  AX.25 summary). VERIFIED live: it connects via the profile and streams
  decoded beacons.
- **Build change:** the venv now needs `--system-site-packages` for the
  system `python3-dbus`/`python3-gi`. Recreated the venv that way;
  documented in requirements.txt, CLAUDE.md, and docs/PROTOCOL.md §3.
  `pytest tests/` still 8/8 green.
- **Docs:** docs/PROTOCOL.md gets a new §3 "Reproducing the Live KISS
  Link — the working method" (full recipe + gotchas), the old §2
  raw-socket/channel-1 guidance marked SUPERSEDED, sections renumbered
  (AX.25 §4, APRS §5, Callsign §6, Legal §7). ROADMAP Phase 2 closed.
- **Known follow-ups (Phase 3):** payloads are **Mic-E** encoded, so
  `aprs.py` must decode Mic-E (destination-address lat + `` ` ``/`'` info
  type), not just standard `!`/`=`/`:`. Also: the transport shows one
  brief disconnect/reconnect right after first connect (the
  Disconnect-first clears state, then it stabilizes) — acceptable, but
  worth smoothing later so the initial connect doesn't flap once.
- **TX WORKS TOO (same session): transmitted an APRS message off-air.**
  Built an AX.25 UI frame by hand (KC3SMW-7 > APZUVT, direct/no path,
  info `:KC3SMW-9 :hello from UVProTermBT{1`), KISS-framed it, sent it via
  `link.send()`. direwolf (monitoring off-air) decoded it cleanly:
  `KC3SMW-7>APZUVT::KC3SMW-9 :hello from UVProTermBT{1` — parsed as an
  APRS Message, number "1". So the link is fully bidirectional: RX (live
  Mic-E beacons + a message ack) and TX both confirmed on real RF. Note
  the direwolf-TX-fails caveat in PROTOCOL.md §2 does NOT apply — that's
  direwolf transmitting; here direwolf only receives, and WE transmit via
  the UV-Pro, which works.
- **Folded the validated encoder into `uvprotermbt/ax25.py`** (first real
  Phase 3 module): `encode_ui_frame()` (control 0x03 / PID 0xF0) plus
  `decode_frame()` for the address field (source > dest via path) and info.
  `Address` dataclass with proper SSID-byte handling — including the
  subtlety that bit 7 is the command/response (C) bit on dest/source but
  the has-been-repeated (H) bit on digipeaters (caught by a failing test
  where an encoded dest round-tripped as "repeated").
- **Tests: `tests/test_ax25.py` (7 tests, all green; 15 total with KISS).**
  Anchored with two real byte vectors: the exact frame direwolf accepted
  on-air (regression guard on the encoder) and a real received Mic-E
  beacon (decode guard). `scripts/monitor.py` now uses `ax25.decode_frame`
  for its summary instead of duplicating the logic.
- **Community guide + standalone demo written and validated.** Added
  `docs/UVPRO_N76_KISS_LINUX.md` (shareable, app-agnostic writeup of the
  whole method: why raw sockets fail, the SerialPort-profile approach,
  pairing/stale-key fix, troubleshooting table, credits) and
  `scripts/uvpro_kiss_demo.py` (self-contained, dependency-free-beyond-
  system-BlueZ-bindings RX/TX reference). Both RX and TX of the demo
  script VERIFIED live with direwolf awake: RX decoded live Mic-E beacons;
  TX sent `KC3SMW-7>APZUVT:>...` decoded off-air as a Status Report.
  Debugging the demo surfaced two things every user will hit, now handled
  in the script + documented: `AutoConnect: True` is mandatory on the
  profile registration (else NewConnection never fires when the desktop

  connect can fail with Errno 103 due to the initial disconnect flap —
  retry on reconnect. (Also a testing note: direwolf's receiver can go to

  reference beacons before concluding a TX didn't go out.)

2026-07-14 — Phase 5: AX.25 connected mode (BBS/Winlink terminal)
- **`ax25_conn.py`: connected-mode core, pure and event-driven.** Control-
  field encode/decode (mod-8 I/S/U frames, P/F bit), connected-mode frame
  builders (SABM/DISC/UA/DM/RR/RNR/REJ/I), and `Ax25Connection` — a
  stop-and-wait (window=1) state machine. No I/O, threads, or wall-clock:
  connect()/send()/disconnect()/on_receive()/on_timer() each return a Result
  (frames to transmit, data to deliver up, state events); the caller owns the
  link and the T1 timer. Keeps it fully unit-testable per the ground rule.
  - Refactored `ax25.py` to expose `build_address_field(dest, source, path,
    command=)` (command frames set dest C=1/source C=0; responses flip) and
    made it accept digipeaters as Address OR (call, ssid) tuples — the tuple
    bug bit both the APRS and connected paths, fixed once at the root.
- **`tests/test_ax25_conn.py` (8):** control-field known values (SABM=0x3F
  etc.) + round-trips, connect handshake, data+RR ack, window-1 queueing,
  disconnect handshake, T1 retransmit-then-give-up (N2=3), out-of-sequence
  I-frame → REJ. Full suite now **30/30**.
- **Wired into the GUI BBS/Winlink tabs.** RX routing now splits UI frames
  (control 0x03 → APRS) from connected-mode frames (→ the active session,
